# ТЗ: умное автоскрытие пустых строк сверху и снизу таблицы расписания

## 1. Цель

Сделать веб-таблицу расписания более компактной по вертикали: автоматически скрывать пустые временные строки **только сверху и снизу** каждой таблицы здания, исходя из фактически видимых занятий.

Улучшение должно быть чисто клиентским и непостоянным: оно не должно менять `base_schedule.json`, `individual_lessons.json`, серверные API, Excel-экспорт, сохранение данных расписания или исходную генерацию данных. Допускается менять клиентский JS/CSS и порядок подключения JS-модулей.

## 2. Контекст проекта и текущая структура

Проект генерирует HTML-расписание с двумя таблицами зданий, например:

```html
<h2>Здание: Villa</h2>
<div class="schedule-container" data-building="Villa">
  <table class="schedule-grid">...</table>
  <div class="activity-block" ...></div>
</div>

<h2>Здание: Kolibri</h2>
<div class="schedule-container" data-building="Kolibri">...</div>
```

Ключевые существующие сущности:

- строки таблицы находятся в `.schedule-grid tbody tr`;
- каждая строка содержит ячейки `td[data-row]`; значение `data-row` — индекс временной строки;
- временная сетка задается глобальными JS-переменными `gridStart` и `timeInterval`, которые эмитируются в `html_javascript.py`;
- блоки занятий имеют класс `.activity-block` и runtime-атрибуты:
  - `data-day`, например `Mo`, `Di`;
  - `data-building`, например `Villa`, `Kolibri`;
  - `data-start-row`;
  - `data-row-span`;
  - `data-lesson-type`;
- позиционирование блоков выполняет глобальная функция `window.updateActivityPositions()` из `js_modules/position.js`;
- видимость дней переключается кнопками `.toggle-day-button` и функцией `window.toggleDay(btn, dayCode)` из `js_modules/core.js`;
- важная особенность текущей логики: у `.toggle-day-button.active` день считается **скрытым**, а не выбранным;
- фильтр типов занятий находится в `js_modules/lesson_type_filter.js`; скрытые фильтром блоки получают класс `.lesson-type-filter-hidden`, при этом CSS скрывает их через `visibility: hidden`, а не через `display: none`;
- режим быстрого добавления включается кнопкой `#toggle-add-mode`; при включении кнопка получает класс `.active`;
- поиск по расписанию находится в `static/schedule_search_ui.js` и уже умеет скрывать строки/колонки собственными классами. Новое автоскрытие строк не должно конкурировать с поиском.

## 3. Основное поведение

### 3.1. Расчет выполняется независимо для каждого здания

Для каждого `.schedule-container[data-building]` диапазон видимых временных строк рассчитывается отдельно.

Пример: если в `Villa` видимые занятия начинаются в 14:30, а в `Kolibri` — в 09:00, таблицы должны иметь разные вертикальные диапазоны.

### 3.2. Учитываются только видимые дни

Расчет должен учитывать только дни, которые сейчас отображаются после переключателей `+/- Mo`, `+/- Di` и т.д.

День считать скрытым, если выполняется хотя бы одно условие:

- существует кнопка `.toggle-day-button.active[data-day="<day>"]`;
- в соответствующей таблице заголовки/ячейки дня фактически скрыты через `display: none`.

День считать видимым, если он не скрыт указанными механизмами.

### 3.3. Учитываются фильтры типов занятий

При расчете компактного диапазона учитывать только блоки занятий, которые не скрыты текущим фильтром типов занятий.

Блоки с классом `.lesson-type-filter-hidden` не должны влиять на расчет верхней и нижней границы.

Пример: если включен фильтр “только индивидуальные”, групповые занятия не должны удерживать таблицу раскрытой по вертикали.

### 3.4. Учитываются только `.activity-block`

Для расчета занятости использовать только блоки занятий `.activity-block`.

Не нужно анализировать содержимое ячеек таблицы, текстовые метки времени, служебные элементы, кнопки добавления/удаления колонок, подсветку ячеек или любые будущие декоративные элементы.

### 3.5. Скрываются только пустые строки сверху и снизу

Нельзя скрывать пустые промежутки внутри диапазона между первым и последним видимым занятием.

Пример: если в здании есть видимые занятия `09:00–11:00` и `17:00–18:00`, таблица должна показывать весь диапазон от начала первого занятия до конца последнего занятия с учетом отступов. Промежуток `11:00–17:00` должен остаться видимым.

## 4. Алгоритм расчета диапазона

### 4.1. Базовые определения

Для каждого здания собрать все “учитываемые блоки”. Блок учитывается, если:

1. он находится внутри текущего `.schedule-container`;
2. его `data-day` относится к видимому дню;
3. он не имеет класса `.lesson-type-filter-hidden`;
4. он имеет валидные координаты строк.

Координаты строк:

- `startRow = parseInt(block.getAttribute('data-start-row'), 10)`;
- `rowSpan = parseInt(block.getAttribute('data-row-span'), 10)`;
- `endRow = startRow + rowSpan`.

Валидные координаты:

- `startRow >= 0`;
- `rowSpan > 0`;
- `endRow > startRow`;
- `startRow` и `endRow` должны попадать в разумные границы таблицы. Если `endRow` равен индексу граничной строки после последнего интервала, это допустимо.

Если у видимого блока отсутствуют `data-start-row` или `data-row-span`, можно использовать существующие fallback-функции `deriveStartRowFromBlock()` и `deriveRowSpanFromBlock()` из `position.js`, если они доступны. Если координаты все равно не удалось определить, см. раздел “Fail-open поведение”.

### 4.2. Отступы

Использовать константу:

```js
COMPACT_ROW_PADDING = 3;
```

Это означает три пустые временные строки до первого занятия и три пустые строки после последнего занятия, если такие строки существуют в сетке.

Верхняя граница:

```js
firstVisibleRow = Math.max(0, minStartRow - COMPACT_ROW_PADDING);
```

Нижняя граница:

```js
lastVisibleRow = Math.min(maxRow, maxEndRow + COMPACT_ROW_PADDING - 1);
```

Здесь `maxEndRow` — максимальный `startRow + rowSpan` среди учитываемых блоков. `maxEndRow` является первой строкой после последнего занятого интервала, поэтому `+ COMPACT_ROW_PADDING - 1` оставляет ровно три пустые строки после блока.

Пример при `timeInterval = 5`:

- первое занятие начинается в 14:30;
- `COMPACT_ROW_PADDING = 3`;
- таблица начинается с 14:15.

Пример для нескольких дней:

- видны понедельник и вторник;
- в понедельник первое занятие начинается в 14:30;
- во вторник первое занятие начинается в 13:00;
- базовая граница — 13:00;
- с учетом отступа 3 строки при `timeInterval = 5` фактическое отображение начинается с 12:45.

Пример для нижней границы:

- последнее занятие заканчивается в 18:00;
- при `timeInterval = 5` остаются видимыми три пустые строки после занятия: 18:00, 18:05, 18:10;
- длинный пустой хвост до 20:00 скрывается.

### 4.3. Строки для скрытия

Для каждой строки таблицы определить ее `rowIndex` через `td[data-row]` внутри этой строки.

Скрывать строку, если:

```js
rowIndex < firstVisibleRow || rowIndex > lastVisibleRow
```

Не скрывать `thead`; заголовок таблицы должен оставаться видимым.

Не скрывать/показывать отдельные ячейки дней напрямую. Новая логика должна управлять только строками `tbody tr` через собственный CSS-класс.

## 5. Поведение при отсутствии занятий

Если для здания нет ни одного учитываемого блока, таблицу нельзя скрывать полностью.

В этом случае нужно показывать минимальный диапазон строк и комментарий о том, что занятий нет.

### 5.1. Минимальный диапазон

Использовать константу:

```js
COMPACT_EMPTY_MINUTES = 60;
```

Если нет учитываемых блоков, показывать первые строки сетки от `gridStart` примерно на один час:

```js
emptyRowsToShow = Math.max(1, Math.ceil(COMPACT_EMPTY_MINUTES / timeInterval));
firstVisibleRow = 0;
lastVisibleRow = Math.min(maxRow, emptyRowsToShow - 1);
```

Если фактическая сетка короче одного часа, показывать всю доступную сетку.

### 5.2. Комментарий

Для здания без учитываемых занятий показывать ненавязчивое сообщение, например:

```text
Нет занятий для выбранных дней и текущего фильтра в здании Villa.
```

Сообщение можно разместить перед соответствующим `.schedule-container`, рядом с заголовком здания или в общей служебной области над расписанием. Предпочтительно делать сообщение отдельным элементом на каждое здание, чтобы было понятно, к какой таблице оно относится.

Требования к сообщению:

- должно появляться только когда автоскрытие активно;
- не должно появляться при активном поиске;
- не должно появляться в режиме добавления;
- должно обновляться при переключении дней и фильтров;
- должно иметь `aria-live="polite"` или аналогичную доступную семантику;
- не должно сохраняться как серверное состояние.

Если скрыты все дни, сообщение должно отражать именно это, например:

```text
Все дни скрыты для здания Villa.
```

## 6. Условия временного отключения автоскрытия

Новое автоскрытие должно иметь приоритеты состояний.

### 6.1. Активный поиск

Если поиск по расписанию активен, новое автоскрытие должно быть полностью отключено.

Активность поиска определять через существующий API, если он доступен:

```js
window.ScheduleSearch && window.ScheduleSearch.isActive && window.ScheduleSearch.isActive()
```

При активном поиске:

- удалить все классы автоскрытия строк;
- скрыть сообщения “нет занятий”;
- не применять компактный диапазон;
- дать `static/schedule_search_ui.js` самостоятельно управлять видимостью строк/колонок;
- после снятия поиска снова пересчитать компактные диапазоны.

Важно: поиск уже использует свои классы, например `.schedgen-search-hidden-row`. Новый код не должен добавлять конкурирующие классы в это время.

### 6.2. Режим добавления `#toggle-add-mode`

Если оператор включил режим добавления через кнопку `#toggle-add-mode`, все временные строки должны сразу отображаться.

Активность режима добавления определять локально по кнопке:

```js
document.getElementById('toggle-add-mode')?.classList.contains('active')
```

При активном add-mode:

- удалить все классы автоскрытия строк;
- скрыть сообщения “нет занятий”;
- оставить скрытыми те дни, которые пользователь скрыл кнопками `+/- Mo`, `+/- Di` и т.д.;
- не раскрывать скрытые колонки/дни;
- не менять серверное состояние;
- не отправлять никаких событий на сервер;
- не влиять на другие браузеры/машины пользователей.

При выключении add-mode компактный диапазон должен пересчитаться заново.

### 6.3. Приоритеты

Приоритеты состояний:

1. активный поиск — автоскрытие отключено;
2. активный add-mode — автоскрытие отключено;
3. обычный режим — автоскрытие включено.

Если поиск и add-mode по какой-то причине активны одновременно, результат один: все временные строки раскрыты, а поиск не должен получать конкурирующие классы автоскрытия.

## 7. Fail-open поведение и критические edge-кейсы

Безопасность отображения важнее компактности.

### 7.1. Невалидные координаты видимого блока

Если в здании есть блок, который должен учитываться по дню и фильтру, но его координаты строк не удалось определить или они явно невалидны, для этого здания нужно временно отключить автоскрытие и показать полную таблицу.

Нужно вывести `console.warn` с указанием проблемы и блока.

Нельзя скрывать строки в контейнере, если есть риск скрыть видимое занятие из-за битых `data-start-row`/`data-row-span`.

### 7.2. Блок за пределами сетки

Если блок выходит за пределы таблицы:

- если координаты можно безопасно зажать в границы таблицы без потери видимого блока, допускается зажимать;
- если невозможно понять корректную границу, использовать fail-open для здания.

### 7.3. Все дни скрыты

Если пользователь скрыл все дни:

- таблица не должна полностью исчезать из-за новой логики;
- показывать минимальный диапазон строк;
- показывать сообщение вроде “Все дни скрыты для здания Villa.”;
- кнопки дней должны оставаться доступными, чтобы пользователь мог снова раскрыть день.

### 7.4. В одном здании занятий нет, в другом есть

Каждое здание обрабатывается независимо.

Если в `Villa` нет учитываемых занятий, а в `Kolibri` есть, `Villa` показывает минимальный диапазон и сообщение, а `Kolibri` показывает рассчитанный компактный диапазон.

### 7.5. Внутренние пустоты не скрывать

Даже если между занятиями большой перерыв, строки внутри рассчитанного диапазона остаются видимыми.

### 7.6. Роли пользователей

Автоскрытие должно работать для всех ролей: `viewer`, `organizer`, `editor`, `admin`.

Разница только в том, что кнопка `#toggle-add-mode` доступна не всем ролям. Если кнопки нет, автоскрытие продолжает работать, а состояние add-mode считается выключенным.

### 7.7. Локальность состояния

Включение add-mode одним пользователем в одном браузере не должно раскрывать строки у других пользователей и не должно записываться на сервер.

### 7.8. Экспорт в Excel

Excel-экспорт должен оставаться полным и не зависеть от визуально скрытых строк.

Новая логика не должна:

- удалять строки из DOM;
- удалять или скрывать `.activity-block`;
- менять `data-start-row`, `data-row-span`, `data-day`, `data-building`, `data-col-index`;
- менять `collectScheduleData()` так, чтобы экспорт стал зависеть от компактного отображения.

Допустимо оставить компактные CSS-классы на строках во время Excel-экспорта, если это не влияет на собираемые данные. Если при тестировании выяснится, что какой-либо экспортный сценарий зависит от геометрии строк, перед экспортом нужно временно снять компактные классы и восстановить их после экспорта.

### 7.9. Сохранение HTML через `outerHTML`

В проекте есть сохранение HTML, где используется `document.documentElement.outerHTML` в `js_modules/save_export.js`.

Так как автоскрытие должно быть непостоянным, перед сериализацией HTML нужно снять классы автоскрытия и служебные сообщения либо реализовать API подготовки к сериализации.

Рекомендуемый API:

```js
window.ScheduleCompactRows.prepareForSerialization(options)
```

Поведение:

- снимает классы `.schedgen-compact-hidden-row`;
- скрывает/удаляет служебные сообщения “нет занятий”;
- при `options.flushLayout === true` вызывает `updateActivityPositions()` после снятия классов;
- возвращает функцию восстановления или объект с методом `restore()`, чтобы текущий экран после сохранения можно было вернуть в компактное состояние.

`save_export.js` должен использовать этот API в местах, где формируется `outerHTML`.

## 8. Рекомендуемая реализация

### 8.1. Новый JS-модуль

Добавить новый модуль:

```text
js_modules/compact_rows.js
```

Модуль должен экспортировать глобальный объект:

```js
window.ScheduleCompactRows = {
  init: initCompactRows,
  refresh: refreshCompactRows,
  clear: clearCompactRows,
  isSuspended: isCompactRowsSuspended,
  prepareForSerialization: prepareCompactRowsForSerialization
};
```

Также желательно экспортировать глобальную функцию для совместимости с текущим стилем проекта:

```js
window.initCompactRows = initCompactRows;
```

### 8.2. CSS-классы

Модуль должен использовать собственные классы, не пересекающиеся с поиском:

```css
.schedgen-compact-hidden-row {
  display: none !important;
}

.schedgen-compact-empty-note {
  margin: 8px 0;
  padding: 8px 10px;
  font-size: 13px;
  border-radius: 6px;
  background: #fff8e1;
  border: 1px solid #ffe082;
  color: #5d4a00;
}

.schedgen-compact-empty-note.is-hidden {
  display: none !important;
}
```

CSS можно внедрить через JS-модуль один раз, чтобы функциональность работала и в сгенерированном `schedule.html`, и в веб-версии `/schedule`.

Не использовать классы поиска `.schedgen-search-*`.

### 8.3. Подключение модуля

В `html_javascript.py` добавить `compact_rows` в список JS-модулей.

Рекомендуемый порядок:

- после `lesson_type_filter`, потому что нужно учитывать `.lesson-type-filter-hidden`;
- после `position`, потому что нужен `updateActivityPositions()` и возможные fallback-функции;
- до `app_initialization`, чтобы `initializeApplication()` мог вызвать `initCompactRows()`.

Например в `add_blocks_module_names` добавить:

```python
'compact_rows',
```

между `lesson_type_filter` и `block_content_sync` либо после `quick_add_mode`, но до `app_initialization`. Главное — чтобы к моменту вызова `initializeApplication()` функция `initCompactRows` уже существовала.

### 8.4. Инициализация

В `js_modules/app_initialization.js` вызвать инициализацию после `initLessonTypeFilter()`:

```js
if (typeof initCompactRows === 'function') {
    initCompactRows();
}
```

После первичной инициализации должен произойти первичный расчет компактных диапазонов и затем позиционирование блоков.

Важно: после изменения видимости строк нужно вызвать `window.updateActivityPositions()`, потому что абсолютные позиции `.activity-block` зависят от геометрии видимых ячеек таблицы.

### 8.5. Точки обновления

Компактный диапазон должен пересчитываться после событий, которые меняют набор учитываемых блоков или их координаты.

Обязательные точки:

1. переключение дня в `window.toggleDay(btn, dayCode)`;
2. применение фильтра типов занятий в `applyLessonTypeFilter(filterValue)`;
3. включение/выключение `#toggle-add-mode`;
4. добавление блока;
5. удаление блока;
6. drag & drop блока;
7. изменение времени/длительности блока;
8. vertical resize блока;
9. обновление индивидуальных занятий из `static/individual_ui.js`;
10. обновление базового расписания из `static/base_sync_ui.js`;
11. активация/деактивация поиска в `static/schedule_search_ui.js`.

Рекомендуется сочетать явные вызовы `ScheduleCompactRows.refresh()` в уже существующих местах с `MutationObserver`, который подстрахует динамические изменения `.activity-block`.

### 8.6. MutationObserver

В `compact_rows.js` можно создать `MutationObserver`, который наблюдает за изменениями внутри `.schedule-container` / `document.body`.

Отслеживать:

- добавление/удаление `.activity-block`;
- изменения атрибутов `.activity-block`:
  - `data-day`;
  - `data-building`;
  - `data-start-row`;
  - `data-row-span`;
  - `data-lesson-type`;
  - `data-col-index`.

Не рекомендуется наблюдать за атрибутом `class` глобально, чтобы собственное добавление `.schedgen-compact-hidden-row` не вызывало циклы. Изменения фильтра типов занятий лучше покрыть явным вызовом из `applyLessonTypeFilter()`.

Observer должен использовать debounce через `requestAnimationFrame`, чтобы серия изменений DOM приводила к одному пересчету.

### 8.7. Обновление `toggleDay`

В `js_modules/core.js` после изменения видимости дня нужно вызвать компактный refresh.

Рекомендуемая логика:

```js
if (window.ScheduleCompactRows && typeof window.ScheduleCompactRows.refresh === 'function') {
    window.ScheduleCompactRows.refresh();
} else {
    updateActivityPositions();
}
```

Если оставить существующий `updateActivityPositions()` перед `refresh()`, это допустимо, но приведет к лишнему позиционированию. Предпочтительно, чтобы `refresh()` сам вызывал `updateActivityPositions()` после изменения строк.

### 8.8. Обновление `lesson_type_filter.js`

В конце `applyLessonTypeFilter(filterValue)` после добавления/удаления `.lesson-type-filter-hidden` вызвать:

```js
if (window.ScheduleCompactRows && typeof window.ScheduleCompactRows.refresh === 'function') {
    window.ScheduleCompactRows.refresh();
} else if (typeof window.updateActivityPositions === 'function') {
    window.updateActivityPositions();
}
```

### 8.9. Обновление add-mode

В `quick_add_mode.js` после изменения класса `.active` у `#toggle-add-mode` вызвать `ScheduleCompactRows.refresh()`.

Также сам `compact_rows.js` должен уметь обнаружить состояние кнопки по DOM, чтобы не зависеть от локальной переменной `addBlockMode`.

### 8.10. Интеграция с поиском

В `static/schedule_search_ui.js` нужно уведомлять `ScheduleCompactRows` при изменении активности поиска.

Возможные варианты:

1. в `syncUiState()` после изменения `data-search-active` вызвать `window.ScheduleCompactRows.refresh()`;
2. либо хранить предыдущее значение `active` и вызывать refresh только при изменении `false -> true` или `true -> false`;
3. либо отправлять custom event, который слушает `compact_rows.js`.

Предпочтительно не вызывать refresh на каждый ввод символа без необходимости, но даже при таком варианте refresh должен быть дешевым и debounce-нутым.

При `ScheduleSearch.isActive() === true` метод `ScheduleCompactRows.refresh()` должен вести себя как `clear()`.

### 8.11. Сериализация HTML

В `save_export.js` добавить подготовку compact rows рядом с уже существующей подготовкой поиска.

Примерный подход:

```js
function _prepareCompactRowsForSavedHtml(options) {
    var compact = window.ScheduleCompactRows;
    if (!compact || typeof compact.prepareForSerialization !== 'function') {
        return null;
    }
    return compact.prepareForSerialization(options);
}
```

Перед `document.documentElement.outerHTML`:

```js
var compactRestore = _prepareCompactRowsForSavedHtml({ flushLayout: true });
var htmlContent = document.documentElement.outerHTML;
if (compactRestore && typeof compactRestore.restore === 'function') {
    compactRestore.restore();
} else if (typeof compactRestore === 'function') {
    compactRestore();
}
```

Нужно применить это и для финального сохранения, и для промежуточного сохранения.

## 9. Детали реализации `compact_rows.js`

### 9.1. Псевдокод refresh

```js
function refreshCompactRows() {
  clearCompactRows({ updatePositions: false });

  if (isCompactRowsSuspended()) {
    hideAllEmptyNotes();
    updatePositionsSoon();
    return;
  }

  document.querySelectorAll('.schedule-container').forEach(function(container) {
    applyCompactRowsToContainer(container);
  });

  updatePositionsSoon();
}
```

### 9.2. Suspended state

```js
function isCompactRowsSuspended() {
  return isSearchActive() || isAddModeActive();
}
```

### 9.3. Применение к контейнеру

```js
function applyCompactRowsToContainer(container) {
  var table = container.querySelector('.schedule-grid');
  var rowMap = collectRowMap(table);
  var maxRow = getMaxRow(rowMap);
  var visibleDays = collectVisibleDays(container);

  if (!visibleDays.length) {
    applyEmptyRange(container, rowMap, maxRow, 'Все дни скрыты для здания ...');
    return;
  }

  var result = collectConsideredBlocks(container, visibleDays, maxRow);

  if (result.hasInvalidVisibleBlock) {
    showFullRange(container);
    hideEmptyNote(container);
    return;
  }

  if (!result.blocks.length) {
    applyEmptyRange(container, rowMap, maxRow, 'Нет занятий ...');
    return;
  }

  var minStartRow = Math.min(...result.blocks.map(b => b.startRow));
  var maxEndRow = Math.max(...result.blocks.map(b => b.endRow));

  var firstVisibleRow = Math.max(0, minStartRow - COMPACT_ROW_PADDING);
  var lastVisibleRow = Math.min(maxRow, maxEndRow + COMPACT_ROW_PADDING - 1);

  applyRange(rowMap, firstVisibleRow, lastVisibleRow);
  hideEmptyNote(container);
}
```

### 9.4. Очистка

`clearCompactRows()` должен удалять `.schedgen-compact-hidden-row` со всех строк и скрывать служебные сообщения.

Не должен трогать:

- `.schedgen-search-hidden-row`;
- `.lesson-type-filter-hidden`;
- inline `style.display` у дней;
- `.activity-block`;
- данные расписания.

## 10. Нефункциональные требования

1. Производительность: пересчет должен быть быстрым даже при большом количестве строк и блоков. Использовать один проход по строкам и блокам на контейнер, без вложенных тяжелых запросов DOM в циклах по каждой ячейке.
2. Идемпотентность: повторный вызов `refresh()` без изменений не должен менять результат и не должен накапливать классы/сообщения.
3. Отсутствие гонок: собственные изменения классов строк не должны приводить к бесконечному циклу MutationObserver.
4. Совместимость: код должен работать, если `window.ScheduleSearch` отсутствует, если `#toggle-add-mode` отсутствует, если есть только одно здание, если у здания нет блоков.
5. Клиентская локальность: никаких запросов на сервер, никаких записей в storage, cookies или JSON-файлы.
6. Доступность: сообщения о пустых выборках должны быть читаемыми и не должны ломать клавиатурную навигацию.
7. Визуальная стабильность: не удалять строки из DOM, только управлять CSS-классом `display:none`.

## 11. Критерии приемки

### 11.1. Один видимый день, позднее начало

Дано:

- виден только `Mo`;
- в `Villa` первое видимое занятие начинается в 14:30;
- до 14:30 в `Villa` нет видимых занятий.

Ожидаемо:

- таблица `Villa` начинается с 14:15 при `timeInterval = 5`;
- строки до 14:15 скрыты;
- блоки занятий позиционируются корректно;
- `Kolibri` рассчитывается независимо.

### 11.2. Несколько видимых дней

Дано:

- видны `Mo` и `Di`;
- в `Mo` первое занятие в `Villa` начинается в 14:30;
- в `Di` первое занятие в `Villa` начинается в 13:00.

Ожидаемо:

- базовая верхняя граница выбирается по `Di` — 13:00;
- с учетом трех строк отступа при `timeInterval = 5` таблица начинается с 12:45.

### 11.3. Нижняя граница

Дано:

- последнее видимое занятие в здании заканчивается в 18:00;
- после 18:00 до конца сетки занятий нет.

Ожидаемо:

- длинный хвост до конца сетки скрыт;
- при `timeInterval = 5` остаются три пустые строки после занятия: 18:00, 18:05, 18:10;
- внутренние пустые промежутки между занятиями не скрываются.

### 11.4. Фильтр типов занятий

Дано:

- включен фильтр “только индивидуальные”;
- групповые занятия начинаются в 09:00;
- индивидуальные занятия начинаются в 13:00.

Ожидаемо:

- расчет верхней границы идет по индивидуальным занятиям;
- групповые блоки с `.lesson-type-filter-hidden` не удерживают таблицу раскрытой.

### 11.5. Add-mode

Дано:

- автоскрытие скрыло верхние/нижние строки;
- оператор нажимает `#toggle-add-mode`.

Ожидаемо:

- все временные строки немедленно становятся видимыми;
- скрытые дни остаются скрытыми;
- при повторном нажатии `#toggle-add-mode` компактный диапазон пересчитывается заново;
- это не влияет на другие браузеры/пользователей.

### 11.6. Поиск

Дано:

- автоскрытие активно;
- пользователь вводит запрос в поиск расписания.

Ожидаемо:

- классы автоскрытия снимаются;
- строками управляет только поиск;
- после очистки поиска автоскрытие снова применяется.

### 11.7. Нет занятий

Дано:

- для `Villa` нет учитываемых занятий в выбранных днях и текущем фильтре.

Ожидаемо:

- таблица `Villa` не исчезает полностью;
- показывается минимальный диапазон около одного часа от начала сетки;
- отображается сообщение “Нет занятий для выбранных дней и текущего фильтра в здании Villa.”;
- при включении add-mode сообщение скрывается и все строки раскрываются.

### 11.8. Все дни скрыты

Дано:

- пользователь скрыл все дни.

Ожидаемо:

- таблицы не ломаются;
- показывается минимальный диапазон строк;
- отображается сообщение “Все дни скрыты для здания ...”; 
- кнопки дней остаются доступны для обратного раскрытия.

### 11.9. Невалидный блок

Дано:

- в видимом дне есть `.activity-block` без валидных `data-start-row`/`data-row-span`, и fallback не смог определить координаты.

Ожидаемо:

- для этого здания показывается полная таблица;
- в консоли есть предупреждение;
- занятие не скрывается случайно.

### 11.10. Сохранение и экспорт

Ожидаемо:

- Excel-экспорт содержит полное расписание и не зависит от скрытых строк таблицы;
- промежуточное и финальное HTML-сохранение не фиксируют в `outerHTML` временные классы `.schedgen-compact-hidden-row`;
- после сохранения текущая открытая страница может вернуться в компактный режим.

## 12. Что не входит в задачу

Не нужно:

- скрывать пустые промежутки внутри дня между занятиями;
- менять серверные API;
- менять формат `base_schedule.json` или `individual_lessons.json`;
- менять алгоритмы генерации расписания;
- менять Excel-экспорт, кроме возможной безопасной подготовки DOM, если тесты покажут необходимость;
- сохранять пользовательскую настройку компактности;
- добавлять серверную настройку включения/выключения автоскрытия;
- менять семантику кнопок дней;
- раскрывать скрытые дни при add-mode.

## 13. Итоговое ожидаемое состояние

После реализации веб-расписание автоматически становится компактнее по вертикали:

- пустые строки до первого видимого занятия скрываются;
- пустые строки после последнего видимого занятия скрываются;
- расчет выполняется отдельно для каждого здания;
- расчет учитывает видимые дни и фильтры занятий;
- внутренние пустые промежутки не скрываются;
- add-mode и поиск временно отключают автоскрытие;
- таблица не исчезает полностью при отсутствии занятий;
- экспорт, сохранение и серверное состояние остаются корректными и полными.
